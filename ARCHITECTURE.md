# Global Pot Project — Data Architecture

## Overview

This document defines the canonical data management architecture for the Global Pot Project. It exists to prevent fragmentation of data across storage locations and to ensure that all data ingestion, processing, and curation work is reproducible and auditable. **All data management work should follow this architecture going forward.**

---

## Principles

1. **One canonical location per data type.** No copies, no ambiguity.
2. **Originals are immutable.** Never modify or delete an original image.
3. **Manually curated files are primary data.** Treat them with the same care as originals.
4. **Algorithmic derivatives are disposable.** They can always be regenerated by pipeline scripts.
5. **Everything enters the project through the intake pipeline.** No ad hoc additions.
6. **All code and metadata live in GitHub.** No scripts or CSVs floating on local drives or cloud storage.

---

## Storage Locations

| Data type | Location | Notes |
|---|---|---|
| Original images | Uni OneDrive `/GlobalPotProject/images/originals/` | Immutable |
| Manually cleaned images | Uni OneDrive `/GlobalPotProject/images/cleaned/` | Primary data, versioned |
| Algorithmic derivatives | Uni OneDrive `/GlobalPotProject/images/derived/` | Disposable, regenerable |
| Master metadata CSV | GitHub repo `/data/master_metadata.csv` | Source of truth |
| Intake & pipeline scripts | GitHub repo `/scripts/` | Python |
| Analysis scripts | GitHub repo `/analysis/` | R |
| Manifests | GitHub repo `/data/manifests/` | Auto-generated, committed |

## Master Metadata CSV

The file `data/master_metadata.csv` is the **single source of truth** for the project. Every pot has exactly one row.

### Required columns

| Column | Description |
|---|---|
| `pot_id` | Unique identifier — canonical across all subprojects |
| `original_filename` | Original filename as received |
| `original_path` | Path relative to OneDrive root |
| `cleaned_path` | Path to manually cleaned version (if exists) |
| `date_accessioned` | ISO date added to project |
| `source` | Where the image came from |
| `metadata_complete` | Boolean — is metadata fully populated? |
| `notes` | Free text |

Additional columns for subproject-specific metadata should be appended but never remove or rename existing columns.

---

## Intake Pipeline

When new images arrive, they must be processed via `scripts/intake.py`, which:

1. Copies originals to `OneDrive/GlobalPotProject/images/originals/` (never moves — always copies first, verify, then optionally delete source)
2. Assigns or validates `pot_id`
3. Adds a row to `master_metadata.csv`
4. Updates the manifest
5. Commits changes to GitHub

**No image should enter the project outside this pipeline.**

---

## Manifests

A manifest is a CSV listing every file in a given folder on OneDrive, with checksums. Manifests live in `data/manifests/` and are committed to GitHub. They serve as a lightweight audit trail and allow `validate.py` to detect drift (files present on OneDrive but missing from metadata, or vice versa).

Regenerate manifests by running `scripts/validate.py`.

---

## Image Naming Convention

- Filename = `pot_id` + extension (e.g. `GPP_004521.jpg`)
- Processing stage encoded in folder, not filename
- Derivatives share the same filename as their original, in their respective subfolder

---

## Derivative Images

| Stage | Folder | Regenerable? |
|---|---|---|
| Filtered | `images/derived/filtered/` | Yes — via pipeline script |
| Segmented | `images/derived/segmented/` | Yes — via pipeline script |
| Profiled | `images/derived/profiled/` | Yes — via pipeline script |
| Manually cleaned | `images/cleaned/` | **No — primary data** |

---

## Subprojects and Students

Each subproject should:
- Use the same `pot_id` system — no new ID schemes
- Add columns to `master_metadata.csv` rather than creating separate metadata files
- Have its own subfolder under `analysis/` in the repo
- Not store data outside the canonical OneDrive structure

---

## What NOT to do

- Do not store images in the GitHub repo
- Do not keep metadata in Excel, Google Sheets, or local CSVs outside the repo
- Do not create derivative images manually without documenting the process in a script
- Do not add images directly to OneDrive without running intake.py
